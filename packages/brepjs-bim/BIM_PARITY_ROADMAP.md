# brepjs-bim — BIM Standards Parity Roadmap

> **Status:** v0.1.0 (experimental, write-only, IFC4-aligned via web-ifc 0.0.77)
> **Scope:** Path to defensible IFC4 / buildingSMART parity across four dimensions — element breadth, import/round-trip, data standards, and validation/certification.
> **Audience:** Lead architect + contributors planning the next several PRs.

---

## 1. Executive Summary

brepjs-bim today is a **focused, write-only IFC4 structural exporter** built on a clean BREP-backed geometry pipeline. It credibly covers the P0 occurrence set — `IfcWall`, `IfcSlab`, `IfcBeam`, `IfcColumn` (all with `IfcExtrudedAreaSolid` SweptSolid geometry and `PredefinedType` handling), `IfcDoor`/`IfcWindow` filled into `IfcOpeningElement` via `IfcRelVoidsElement`/`IfcRelFillsElement`, a complete `IfcProject → IfcSite → IfcBuilding → IfcBuildingStorey` spatial hierarchy wired with `IfcRelAggregates` + `IfcRelContainedInSpatialStructure`, and partial `Pset_*Common` + `Qto_*BaseQuantities` per element. For a v0.1 this is a respectable, internally-consistent core.

But "BIM-standards parity" is a much larger claim than what exists. The blunt per-dimension assessment:

- **Element breadth — ~40% of a baseline architectural/structural claim.** The occurrence core is solid, but there is **no type layer** (`IfcWallType`/`IfcDoorType`/`IfcRelDefinesByType`), **no `IfcBuildingElementProxy` catch-all**, **no layered materials** (`IfcMaterialLayerSet`/`Usage`), only **3 of ~11 parametric profiles**, no `IfcSpace`, and none of Roof/Stair/Ramp/Railing/Covering/CurtainWall/Footing/Pile. Geometry is SweptSolid-only (no Brep/Tessellation, no wall Axis). Doors/windows emit `Representation: null`.
- **Import / round-trip — ~0%. Greenfield.** There is no reader, no `fromIfc`, no model-graph access. `tests/roundTrip.test.ts` is a misnomer (it re-opens its own output to count entities). The favorable fact: web-ifc, already a dependency, exposes the full read API, so the build is unblocked — it just doesn't exist.
- **Data standards — ~30%.** Occurrence-level Psets/Qtos exist but are incomplete subsets with **wrong measure types** (every number → `IFCREAL`), no type objects, no classification, no layered/profiled materials, no `IfcSpace/Zone/System`, no MVD declaration, and **non-deterministic GUIDs** that silently break every diff/COBie/update workflow.
- **Validation / certification — ~5%. Effectively from-scratch.** No EXPRESS/STEP conformance gate on own output, no MVD declared, no IDS, no BCF, no geometry-validity gate (despite brepjs owning `BRepCheck`/`autoHeal`), no GUID-uniqueness/referential-integrity checks, no error-vs-warning severity model, and no external-validator CI gate.

The strategic read: the **type layer, deterministic identity, an MVD declaration, and a self-validation/round-trip gate are foundational prerequisites** that nearly every higher-severity item depends on. They are also mostly cheap. Sequencing these first de-risks and unblocks the entire downstream roadmap, including the large import and certification investments.

## 3. Prioritized Phases

Phases are sequenced by dependency and leverage. **Phase 1 is deliberately dependency-free, high-leverage foundation work** that unblocks the rest.

### Phase 1 — Foundations: identity, MVD, type layer, self-validation gate

**Goal:** Make every later conformance claim _meaningful and non-regressing_ before adding breadth.

| Closes                                                                                              | IFC entities                                                                                                         | Effort |
| --------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- | ------ |
| Deterministic GUIDs (stable derivation; stop calling `newIfcGuid()` fresh per Pset/rel)             | `IfcRoot.GlobalId`                                                                                                   | M      |
| GUID uniqueness/collision detection                                                                 | `IfcRoot`                                                                                                            | S      |
| Declare MVD in `FILE_DESCRIPTION` ViewDefinition (`ReferenceView_v1.2`) + `IfcProject` context      | `IfcProject`                                                                                                         | S      |
| `IfcTypeObject` layer + `IfcRelDefinesByType` (Wall/Slab/Beam/Column/Door/Window types)             | `IfcWallType`, `IfcSlabType`, `IfcBeamType`, `IfcColumnType`, `IfcDoorType`, `IfcWindowStyle`, `IfcRelDefinesByType` | L      |
| EXPRESS/STEP self-validation gate on own output                                                     | —                                                                                                                    | L      |
| Write→read→re-write round-trip self-check (using web-ifc's existing read API)                       | —                                                                                                                    | M      |
| Referential-integrity + spatial-containment completeness (no orphaned/duplicate-contained elements) | `IfcRelContainedInSpatialStructure`, `IfcRelAggregates`                                                              | M      |
| Error-vs-warning severity model for validation results                                              | —                                                                                                                    | S→M    |

**Why first:** The type layer is named by both the verified set and the completeness critic as a _foundational prerequisite_ — door/window types, type-level Psets, COBie Type rows, and standard-case modeling all depend on it. Deterministic + unique GUIDs are the identity floor for COBie, BCF, diffing, and any future import-merge. The self-validation + round-trip gate is the cheapest real validation any exporter ships and catches the exact SELECT-serialization class of bug already documented in `ifcWriter.ts`. None of this depends on anything else.

### Phase 2 — Data conformance & geometry-representation breadth

**Goal:** Make the data brepjs-bim already writes _correct and complete_, and unlock arbitrary geometry.

| Closes                                                                                              | IFC entities                                                                                                               | Effort |
| --------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- | ------ |
| Correct Pset measure types (per-property type table, not value-shape heuristic)                     | `IfcThermalTransmittanceMeasure`, `IfcIdentifier`, `IfcLabel`, `IfcPropertySingleValue`                                    | M      |
| Template-driven, complete `Pset_*Common` (+ `IfcPropertyEnumeratedValue` for Status etc.)           | `IfcPropertySet`, `IfcPropertyEnumeratedValue`                                                                             | M      |
| Layered/profiled materials                                                                          | `IfcMaterialLayerSet`, `IfcMaterialLayerSetUsage`, `IfcMaterialLayer`, `IfcMaterialProfileSet`, `IfcRelAssociatesMaterial` | L      |
| Classification references                                                                           | `IfcClassification`, `IfcClassificationReference`, `IfcRelAssociatesClassification`                                        | M      |
| Brep/Tessellation representation + wall **Axis** (reuse brepjs meshing)                             | `IfcFacetedBrep`, `IfcTriangulatedFaceSet`, `IfcPolygonalFaceSet`                                                          | M      |
| `IfcBuildingElementProxy` catch-all (depends on Brep/Tessellation)                                  | `IfcBuildingElementProxy`, `IfcBuildingElementProxyTypeEnum`                                                               | M      |
| Geometry-validity gate on export (closed/manifold/non-self-intersecting via `BRepCheck`/`autoHeal`) | `IfcSolidModel`                                                                                                            | M      |
| Door/Window real geometry + `OverallHeight`/`OverallWidth` (fix `Representation: null`)             | `IfcDoor`, `IfcWindow`, `IfcDoorLiningProperties`, `IfcWindowPanelProperties`                                              | M      |

**Why second:** Depends on Phase 1's type layer (type-level Psets/materials) and severity model. The Brep/Tessellation path is itself a prerequisite for the proxy element and for furnishing/MEP later, and for the tessellated _import_ fallback in Phase 4. Fixing measure types + completing templates is required for any IDS/validator pass.

### Phase 3 — Element & profile breadth

**Goal:** Expand the authorable taxonomy to a credible architectural/structural set.

| Closes                                                                                                | IFC entities                                                                                                                                                                                                                                                      | Effort |
| ----------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| Parametric profile breadth (L/T/U/Z/C, asymmetric I, ellipse, trapezium, hollow)                      | `IfcLShapeProfileDef`, `IfcTShapeProfileDef`, `IfcUShapeProfileDef`, `IfcZShapeProfileDef`, `IfcCShapeProfileDef`, `IfcAsymmetricIShapeProfileDef`, `IfcEllipseProfileDef`, `IfcTrapeziumProfileDef`, `IfcRectangleHollowProfileDef`, `IfcCircleHollowProfileDef` | M      |
| Arbitrary closed profiles (substitutes for many missing parametrics)                                  | `IfcArbitraryClosedProfileDef`, `IfcArbitraryProfileDefWithVoids`                                                                                                                                                                                                 | M      |
| `IfcElementAssembly` / element-level `IfcRelAggregates` / `IfcRelNests` (prerequisite for composites) | `IfcElementAssembly`, `IfcRelAggregates`, `IfcRelNests`                                                                                                                                                                                                           | M      |
| Core architectural elements                                                                           | `IfcRoof`, `IfcStair`, `IfcStairFlight`, `IfcRamp`, `IfcRampFlight`, `IfcRailing`, `IfcCovering`                                                                                                                                                                  | XL     |
| Curtain wall / facade & light framing                                                                 | `IfcCurtainWall`, `IfcPlate`, `IfcMember`                                                                                                                                                                                                                         | L      |
| Foundations                                                                                           | `IfcFooting`, `IfcPile`                                                                                                                                                                                                                                           | M      |
| `IfcSpace` + space boundaries (also unblocks COBie Space)                                             | `IfcSpace`, `IfcSpatialZone`, `IfcRelSpaceBoundary`                                                                                                                                                                                                               | L      |
| Presentation/styling + connectivity (visual usability, structural joins)                              | `IfcStyledItem`, `IfcSurfaceStyle`, `IfcPresentationLayerAssignment`, `IfcRelConnectsElements`, `IfcRelConnectsPathElements`                                                                                                                                      | M      |

**Why third:** Profiles and arbitrary-profile support feed curtain-wall/member realism. Composite elements (stairs, curtain walls, railings) are _intrinsically aggregated_ and would produce structurally invalid IFC without the `IfcElementAssembly`/`IfcRelAggregates`-on-elements machinery — so assembly support precedes them within the phase. All of these benefit from Phase 1's type layer and Phase 2's representation/Pset infrastructure.

### Phase 4 — IFC import & round-trip

**Goal:** Turn brepjs-bim bidirectional, starting with a true symmetric round-trip of its own output.

| Closes                                                                                              | IFC entities                                                                                                                                           | Effort |
| --------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ------ |
| Core SPF parser / model-graph access (`OpenModel`, `GetLine`, `GetLineIDsWithType`, GUID↔expressID) | `IfcRoot`, `IfcGloballyUniqueId`                                                                                                                       | M      |
| Stable identity on import (preserve incoming `GlobalId`; read `IfcOwnerHistory`)                    | `IfcRoot.GlobalId`, `IfcOwnerHistory`                                                                                                                  | L      |
| Spatial-tree reconstruction                                                                         | `IfcRelAggregates`, `IfcRelContainedInSpatialStructure`                                                                                                | M      |
| Placement-chain composition (+ georeferencing/recentering)                                          | `IfcLocalPlacement`, `IfcAxis2Placement3D`, `IfcMapConversion`, `IfcProjectedCRS`                                                                      | M      |
| Unit parsing on import                                                                              | `IfcUnitAssignment`, `IfcSIUnit`, `IfcConversionBasedUnit`                                                                                             | S      |
| SweptSolid + profile reconstruction (symmetric round-trip)                                          | `IfcExtrudedAreaSolid`, `IfcRevolvedAreaSolid`, `IfcRectangleProfileDef`, `IfcCircleProfileDef`, `IfcIShapeProfileDef`, `IfcArbitraryClosedProfileDef` | M→L    |
| Voids/fills + Psets/Qtos read-back                                                                  | `IfcRelVoidsElement`, `IfcRelFillsElement`, `IfcRelDefinesByProperties`, `IfcPropertySingleValue`                                                      | M      |
| Generic `IfcProduct` ingestion + tessellated mesh fallback (real third-party files)                 | `IfcTriangulatedFaceSet`, `IfcPolygonalFaceSet`, `IfcFaceBasedSurfaceModel`                                                                            | L      |
| Geometric robustness (per-element try/heal/skip + diagnostics)                                      | `BRepCheck_Analyzer`/`ShapeFix`                                                                                                                        | L      |
| Schema branching (IFC2x3 / IFC4 / IFC4.3)                                                           | `GetModelSchema`                                                                                                                                       | L      |
| Multiple-representation selection (Body vs Axis/FootPrint/Box)                                      | `IfcShapeRepresentation.RepresentationIdentifier`, `IfcGeometricRepresentationSubContext`                                                              | M      |
| Worker-offloaded, streaming, lifecycle-managed pipeline for large files                             | `OpenModel`/`CloseModel` + brepjs worker layer                                                                                                         | L      |
| Parametric Clipping/CSG/Brep reconstruction (real Revit/ArchiCAD walls)                             | `IfcBooleanClippingResult`, `IfcBooleanResult`, `IfcHalfSpaceSolid`, `IfcFacetedBrep`, `IfcAdvancedBrep`                                               | XL     |
| STEP string decoding (`\X2\`/`\S\`/`\X\` escapes)                                                   | ISO 10303-21 directives                                                                                                                                | S      |

**Why fourth:** Import is the largest greenfield and depends on Phase 1 identity (GUID preservation) and Phase 2/3 representations/profiles to reconstruct into. Sequence _within_ the phase: parser → identity/units → spatial tree → placement → symmetric SweptSolid round-trip → tessellated fallback → robustness/streaming → Clipping/CSG. The symmetric round-trip is the natural first target because it validates the writer too.

### Phase 5 — Certification, IDS, BCF & FM deliverables

**Goal:** Move from "valid output" to "certifiable, requirement-checkable, collaboration-ready."

| Closes                                                                                                             | IFC entities                                                                                                           | Effort |
| ------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------- | ------ |
| IDS 1.0 parser + checker (6 facets, XSD restrictions)                                                              | —                                                                                                                      | XL     |
| BCF-XML 3.0 read/write (ties to existing `ifcGuid`)                                                                | —                                                                                                                      | M→L    |
| COBie deliverable + `IfcSpace`/`IfcZone`/`IfcSystem` (depends on Phase 3 `IfcSpace`)                               | `IfcSpace`, `IfcZone`, `IfcSystem`, `IfcRelAssignsToGroup`                                                             | L      |
| Real `IfcOwnerHistory` / `IfcPerson` / `IfcOrganization` / contacts (COBie Contact key)                            | `IfcOwnerHistory`, `IfcPerson`, `IfcOrganization`                                                                      | M      |
| Qto weights + member-name verification (needs material density)                                                    | `IfcQuantityWeight`, `IfcElementQuantity`                                                                              | M      |
| Schema-version abstraction (IFC4 ↔ IFC4X3) + per-schema entity guards                                              | —                                                                                                                      | L      |
| Official RV cert fixtures + external-validator (ifcopenshell / bSI Validation Service) CI gate + golden-file diffs | —                                                                                                                      | L→XL   |
| Pset/Qto + bSDD warning-tier validation                                                                            | `IfcPropertySet`, `IfcClassificationReference`                                                                         | M      |
| MEP / distribution branch _(own dimension; defer unless scoped)_                                                   | `IfcDuctSegment`, `IfcPipeSegment`, `IfcDuctFitting`, `IfcAirTerminal`, `IfcDistributionSystem`, `IfcDistributionPort` | XL     |
| Furnishings                                                                                                        | `IfcFurnishingElement`, `IfcFurniture`                                                                                 | M      |
| gbXML / IFC-JSON / IFCXML _(adjacent, only if in scope)_                                                           | —                                                                                                                      | L–XL   |

**Why last:** Certification, IDS, and COBie depend on the type layer, complete/correct Psets, classification, deterministic identity, `IfcSpace`, and the self-validation gate — i.e. nearly everything in Phases 1–3. MEP and gbXML/IFC-JSON are explicitly separate domains and should be gated behind an explicit product decision, not the structural/architectural compliance claim.

## 5. Risks & Open Questions

1. **web-ifc read fidelity.** web-ifc reads IFC-SPF only (no IFC-JSON/IFCXML). `WriteLine` does _not_ enforce WHERE-rules, inverse attributes, or SELECT-type serialization — the documented bug class in `ifcWriter.ts` (lines 41–44). The self-validation gate and round-trip self-check must be treated as primary correctness mechanisms, not nice-to-haves. **Open:** is web-ifc's structural enforcement sufficient, or do we need an IfcOpenShell-based external validator from day one?
2. **BREP ↔ IFC geometry round-trip.** Whether brepjs core can _receive_ a tessellated/triangulated solid (mesh → BREP) is unverified and may add significant work to the import tessellated-fallback and to the Brep/Tessellation export path. Parametric Clipping/CSG import maps onto brepjs booleans but lossless reconstruction of `IfcAdvancedBrep` (NURBS) is the hardest case and may never be fully lossless.
3. **Quantity ↔ geometry consistency.** Quantities are currently computed analytically from spec dimensions, not measured from the actual `ValidSolid`. RV certification and validators check geometry/quantity self-consistency; declaring an MVD while emitting analytically-derived quantities risks failing certified checkers. Decision needed: measure from geometry, or reconcile.
4. **Georeferencing & precision.** Real models carry million-metre eastings/northings; feeding those into OCCT blows up precision. Both export (`IfcMapConversion`/`IfcProjectedCRS`) and import (recentering to a local origin) need a coordinate strategy. **Open:** what is the canonical local-origin convention?
5. **MVD / certification cost.** RV1.2 must be certified before DTV; actual b-cert.org certification is an org/process step layered on top of clean Validation Service passes. **Open:** is the goal certification-_ready_ (clean validator in CI + fixtures) or formal certification? They differ substantially in cost and timeline. Verify the exact RV version string against the live MVD package before hardcoding it.
6. **Identity strategy.** Deterministic GUID derivation must be stable across re-exports _and_ able to carry an _incoming_ GUID on import-merge. The derivation function design is load-bearing for COBie, BCF, and federated coordination — get it right in Phase 1 before downstream code depends on it.
7. **Scope boundaries.** MEP (`IfcDistributionElement` branch), gbXML, and IFC-JSON/IFCXML are explicitly separate domains. They should be gated behind a product decision, not folded into the structural/architectural parity claim.
8. **Schema versioning.** IFC4 is hardcoded. National mandates increasingly require IFC4X3. A schema-version abstraction with per-schema entity guards (e.g. `IfcRailway`/`IfcAlignment` only in 4X3) is needed before certification can target multiple schemas — and retrofitting it later is costly.

These are the cheapest items that unblock or de-risk the most downstream work. Most belong in Phase 1.
